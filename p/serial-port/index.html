普通中断接收

  普通中断接收的话，就不进行详细说明，只说几句重点！

1
2
3
4
5

void BLE_UART_Init(void)
{
    // 开启USART2接收中断
    HAL_UART_Receive_IT(&huart2, &RxData, 1);
}

  常规中断接收，需要调用HAL_UART_Receive_IT(&huart2, &RxData, 1)开启 USART2 接收中断；第二个参数是接收缓冲区，用于存放串口收到的数据，缓冲区的形式由第三个参数决定；第三个参数代表收满多少字节后触发接收完成中断。当前参数为 1，每收到 1 字节就进入一次中断，缓冲区使用单个uint8_t变量；若第三个参数大于等于 2，则需要传入数组作为缓冲区存储多字节数据。该函数不限制只能在主函数调用，初始化、中断回调、自定义函数内均可调用。

补充说明

  1. 串口传输时，每个英文字母、符号都占 1 字节（ASCII 编码）；
  2. 如果你代码里存到数组还额外加了 \0 字符串结束符，内存中会占用 11 字节，但线上传输的帧长度依旧是 10 字节；
  3. 结合你上面的 1 字节接收中断：这一整帧会触发 10 次接收中断，逐字符存入缓存数组。

#include "../Inc/uart_callback.h"
char BlueSerial_RxPacket[100];
uint8_t BlueSerial_RxFlag;
uint8_t RxData;       // 存放接收到的单个字节
static uint8_t RxState = 0;  // 接收状态机，0空闲，1接收中
static uint8_t pRxPacket = 0;// 数据包指针，0空闲，1接收中

/**
 * @brief  UART接收中断回调函数，处理数据包
 * @note  状态机解析[]包裹的数据，全局变量无修改
 */
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart)
{
    if (huart->Instance == USART2)
    {
        if (RxState == 0)
        {
            if (RxData == '[' && BlueSerial_RxFlag == 0)
            {
                RxState = 1;
                pRxPacket = 0;
            }
        }
        else if (RxState == 1)
        {
            if (RxData == ']')
            {
                RxState = 0;
                BlueSerial_RxPacket[pRxPacket] = '\0';
                BlueSerial_RxFlag = 1;
            }
            else
            {
                BlueSerial_RxPacket[pRxPacket] = RxData;
                pRxPacket ++;
                // 防止数组越界截断
                if (pRxPacket >= 99) pRxPacket = 99;
            }
        }

        // 重新开启单次接收中断
        HAL_UART_Receive_IT(&huart2, &RxData, 1);
    }
}

DMA配合中断接收

  本章节其实主要是想来讲一下DMA配合串口进行中断接收。当然，上面讲述的常规中断就能实现对小车蓝牙串口的数据解析。而且 HAL_UART_Receive_IT(&huart2, &RxData, 1);的最后一个参数为1，这样每接收到一个字节就会进一次中断，况且有帧头帧尾，这样也可以实现不定长数据接收，这在当前的平衡车是行的通的。但每接收一个字节就让CPU处理一下，若在数据发送频率很高时，CPU也就太忙了，可能会导致CPU处理其他数据时处理不过来，所以，在我平衡车蓝牙遥控效果不太理想时，我尝试了利用DMA进行不定长数组的接收与处理。虽然现在已经证实蓝牙串口并不是蓝牙遥控控制效果不好的主要决定因素，就算不使用DMA，平衡车也能实现较好的蓝牙遥控,而且在我改完小车大部分BUG后，小车能实现直立和速度环控制效果正常后，我又试着用DMA，结果发现控制效果比没使用DMA的控制效果差，具体原因目前还没发现，所以，接下来的DMA配合中断接收，也许不适合用在平衡车上，也也许存在些许BUG，若又发现什么不对，请在评论区告知一声，若评论区使用不了的话，请发邮箱告诉我！谢谢！邮箱：3152067902@qq.com。

void BLE_UART_CopyData(uint8_t Size)
{
    if (Size < 3) return;  // 防下溢（越界 / 非法拷贝）。江协数据包格式>=4字节
    memcpy(BLEProcessed_Data, BLERxInitialData + 1, Size - 2);
    BLEProcessed_Data[Size - 2] = '\0';  // 加终止符，字符串处理函数读到`\0`才停止
}

因为当前采用DMA进行不定长数据的接收，所以会把帧头帧尾一同储存，所以要进行对原始数据的处理。if (Size < 3) return; // 防下溢与BLEProcessed_Data[Size - 2] = '\0'; // 加终止符必须有，否则会出现问题的。

第2小节存在的BUG 原因分析：数据竞争（Race Condition）

根本问题

DMA 串口接收中断和主循环同时访问同一个缓冲区 BLEProcessed_Data，没有任何保护机制。

问题链路图 

1
2
3
4
5
6
7

DMA中断写入 BLEProcessed_Data
         ↓
主循环正在用 strtok 解析 BLEProcessed_Data
         ↓
DMA中断再次触发，覆盖了正在解析的数据
         ↓
strtok 解析到垃圾值 → 遥控目标值被意外修改 → 控制效果变差

具体表现

问题 后果
BLEProcessed_Data 被 DMA 中断覆盖 主循环解析到不完整的数据
strtok 不是线程安全的 中断打断解析状态，导致越界读取
遥控目标值被错误修改 目标错误，平衡车控制失常

解决方案：双缓冲 + 临界区保护

核心思路：中断中写入数据，主循环中读取数据，必须使用临界区或双缓冲来保护共享资源。

uint8_t BLEBackupBuffer[150];  // 备份缓冲区

void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
    if (huart->Instance == USART2) {
        BLE_UART_CopyData(Size);  // 去帧头帧尾

        // 关键：立即拷贝到备份缓冲区
        memcpy(BLEBackupBuffer, BLEProcessed_Data, Size);

        HAL_UARTEx_ReceiveToIdle_DMA(&huart2, BLERxInitialData, sizeof(BLERxInitialData));
        __HAL_DMA_DISABLE_IT(&hdma_usart2_rx, DMA_IT_HT);

        BlueSerial_RxFlag = 1;
    }
}

工作流程总结 

1

DMA中断收到数据
    ↓
写入 BLEProcessed_Data → 拷贝到 BLEBackupBuffer
                              ↓
主循环读取 ← 关闭DMA中断 ← 从BLEBackupBuffer拷贝到LocalBuffer
    ↓
开启DMA中断 → 用 strtok 解析 LocalBuffer（本地变量，安全）

这样 DMA 中断和主循环不再共享同一个正在被修改的缓冲区，数据竞争问题彻底解决。
